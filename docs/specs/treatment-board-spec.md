# מסמך אפיון: לוח הטיפול — מודל סטטוס לשלב הטיפול במלר"ד

**גרסה:** 0.1 (טיוטה, בעבודה) · **תאריך:** 2026-06-17 · **סטטוס:** בעבודה
**קהל יעד:** הצוות המפתח (להזנה ל-plan mode) + הצגה למנהל המוקד
**ענף עבודה:** `feature/treatment-board`

> מסמך זה מאפיין את **השלב האמצעי** בזרימת המטופל (קבלה → **טיפול** → שחרור).
> הקבלה והשחרור ברורים; הטיפול הוא לא-לינארי וקשה לייצוג. זה הליבה.

---

## 1. מבט-על ומטרה

היום לוח התור מציג **סטטוס גס יחיד** מתוך 5 (`VisitStatus`: Waiting/Called/InTreatment/FinishedTreatment/Discharged), בעוד שבמציאות מטופל בשלב הטיפול נמצא בו-זמנית בכמה מצבים (ממתין לתוצאות מעבדה, חזר מצילום, ממתין לייעוץ, אצל אחות) **ללא סדר קבוע**. כל העושר הזה קבור היום בטבלת "ניתוב/הפניות" שבתוך הטופס ואינו מיוצג בלוח.

**המטרה:** מודל סטטוס + לוח שמציגים את המציאות **בפירוט שימושי** ונשארים **סריקים ופשוטים** — איש צוות צריך להבין במבט אחד על שורה אחת: באיזה שלב המטופל, אצל מי הוא כרגע, ועל מה מחכים.

**קריטריון הצלחה:** רופא/אחות מזהים ממבט בשורה בודדה את השלב + הגורם המטפל הנוכחי + הפעילויות הפתוחות, בלי לפתוח את הטופס; והלוח לא "מתפוצץ" לעשרות סטטוסים.

**בהיקף (v1):**
- מודל הסטטוס בן 3 הצירים (להלן §3–§4).
- ייצוג הלוח (§5).
- מעברים והרשאות — מי רשאי לקדם שלב / לנהל פעילויות / להחליף גורם מטפל (§7–§8).

**מחוץ להיקף (לא-יעדים):**
- אינטגרציה למערכת מעבדה/PACS אמיתית (תוצאות מוזנות ידנית ע"י צוות).
- שינוי שלב הקבלה והשחרור עצמם (נשארים כפי שהם, מלבד נקודות החיבור).
- ניהול מיטות פיזי/מפה של המחלקה.

> **גזירה מודעת:** §6 (אינטגרציות חיצוניות) כמעט-ריק בכוונה — המערכת On-Prem ללא אינטגרציות בשלב זה.

## 2. אקטורים ותפקידים

| תפקיד | אימות | מה מותר לו בשלב הטיפול |
|-------|-------|------------------------|
| קבלה (Reception) | מאומת | קבלה (יוצר ביקור → שלב "ממתין לטריאז׳"), שחרור/תשלום. **אין** גישה קלינית. |
| אחות (Nurse) | מאומת | טריאז׳, מדדים, ביצוע פעילויות (מתן/השגחה), עדכון מצב פעילות, החלפת "אצל מי" (קבלת/מסירת מטופל). |
| רופא (Doctor) | מאומת | טיפול, הזמנת פעילויות, החלטת דיספוזיציה, **חתימה** → "סיים טיפול", הפניה/אשפוז. |
| מנהל משמרת (ShiftManager) | מאומת | פיקוח; תיקון שלב/הקצאה; חלון עריכה לאחר חתימה (קיים). |
| אדמין (Admin) | מאומת | הכול (תפעול/תיקון). |

## 3. תזרים העבודה — מודל 3 הצירים

המפתח: **"סטטוס" אינו ערך יחיד**, אלא שלושה צירים אורתוגונליים + דגלים. כך נתפסת אי-הלינאריות בלי פיצוץ סטטוסים.

### ציר 1 — שלב (Phase): ערך יחיד, כמעט-מונוטוני (השדרה)
```
ממתין לטריאז׳ → בטריאז׳ → ממתין לרופא → בטיפול → סיים טיפול (ממתין לשחרור) → שוחרר
                                                         ↘ הופנה/אושפז
   (עזב ללא טיפול — LWBS: מצב סיום חלופי מכל שלב מוקדם)
```
מיפוי ל-`VisitStatus` הקיים: `Waiting→ממתין לטריאז׳/לרופא`, `Called→ממתין לרופא`, `InTreatment→בטיפול`, `FinishedTreatment→סיים טיפול`, `Discharged→שוחרר`.

**החלטה (§10 פתוחה): "דיספוזיציה"** — מקופלת לתוך "בטיפול" (הרופא מחליט, והחתימה היא המעבר ל"סיים טיפול"), אלא אם תרצה שלב "ממתין להחלטה" נפרד.

### ציר 2 — פעילויות פתוחות (Activities): אפס או יותר, מקבילות
כל פעילות עם מצב משלה. **ברירת-מחדל מוצעת: 3 מצבים** `הוזמן → בביצוע → תוצאה/הושלם` (+ `בוטל`). קטלוג (ממוחזר מ-`STATION_OPTIONS`):
- **בדיקות/הדמיה:** מעבדה · א.ק.ג · רנטגן · US · *(CT?)*
- **ייעוצים/הפניות:** ייעוץ מומחה (עם פירוט תחום) · רופא טראומה · רופאת הריון · מוקד 119
- **טיפול/השגחה:** מתן תרופה/עירוי · השגחה (אובזרבציה)

### ציר 3 — אצל מי כרגע (CurrentOwner / Station): ערך יחיד, זז חופשי
התחנה/הגורם המטפל הנוכחי (רופא ר.דחופה / אחות טיפולים / …). **זה** מה שמשתנה בהלוך-ושוב רופא↔אחות — **לא** השלב.

### דגלים (Flags): אופציונליים, מרובים
בידוד · ממתין לתוצאות · ממתין למיטה · דחוף (לפי טריאז׳).

### דוגמה עובדת — "רופא שלח לאחות" (השאלה ששברה את המודל הפשוט)
| פעולה | שלב | אצל מי | פעילויות |
|---|---|---|---|
| רופא בודק | בטיפול | רופא ר.דחופה | — |
| רופא מזמין מתן תרופה ושולח לאחות | בטיפול | **אחות טיפולים** | מתן תרופה: הוזמן |
| אחות מבצעת | בטיפול | אחות טיפולים | מתן תרופה: בביצוע |
| חוזר לרופא | בטיפול | **רופא ר.דחופה** | מתן תרופה: הושלם |

השלב יציב לאורך כל הזמן; ההלוך-ושוב חי ב-CurrentOwner + הפעילויות.

**מסלולי כשל/חלופה:** ביטול פעילות (בוטל); עזיבה ללא טיפול (LWBS) מכל שלב מוקדם; חזרה לשלב מוקדם (נדיר — רק תיקון ע"י מנהל משמרת, מתועד).

## 4. מודל נתונים ומוסכמות

### ישויות ויחסים
- `Visit` (קיים) — מורחב: `Phase`, `CurrentStation`, `CurrentOwnerUserId?`, `Flags`.
- `VisitActivity` (**חדש**) — 1:N מ-`Visit`. מחליף את ייצוג הפעילויות הפזור.
- היסטוריית "אצל מי" — ממוחזרת מטבלת ה-`RoutingJson` הקיימת (תחנה + ת.הגעה = לוג המעברים); `CurrentStation` = הרשומה האחרונה.

### סכימה: `Visit` (תוספות)
| שדה | טיפוס | חובה? | תיאור |
|-----|-------|-------|-------|
| Phase | enum `VisitPhase` | כן | השדרה (ציר 1). מחליף/עוטף את `VisitStatus`. |
| CurrentStation | string (מתוך קטלוג התחנות) | לא | אצל מי כרגע (ציר 3). |
| CurrentOwnerUserId | Guid? | לא | הקלינאי האחראי כרגע. |
| Flags | string[] / bitflags | לא | דגלים (ציר דגלים). |

### סכימה: `VisitActivity` (חדש)
| שדה | טיפוס | חובה? | תיאור |
|-----|-------|-------|-------|
| Id | Guid | כן | מפתח ראשי |
| VisitId | Guid | כן | FK ל-Visit |
| Type | enum `ActivityType` | כן | Lab/ECG/Xray/US/CT/Consult/Medication/Observation/Referral |
| Detail | string? | לא | פירוט (תחום ייעוץ, שם תרופה…) |
| State | enum `ActivityState` | כן | Ordered → InProgress → Done (+ Canceled) |
| OrderedByUserId | Guid | כן | מי הזמין |
| OrderedAt | DateTime | כן | מתי הוזמן |
| UpdatedByUserId | Guid? | לא | מי עדכן מצב אחרון |
| ResultedAt | DateTime? | לא | מתי התקבלה תוצאה/הושלם |
| Note | string? | לא | הערה חופשית/תוצאה תמציתית |

### מוסכמות
- **enum-ים נשמרים כמחרוזת** (עקבי עם `VisitStatus`/`UserRole` הקיימים → קריא ב-DB, עמיד לשינוי סדר).
- ערכי `Phase` הקיימים ב-`VisitStatus` עוברים מיגרציה לערכי `VisitPhase` המקבילים.

## 5. אפיון UI — לוח הטיפול

**מלאי המסכים:** (1) לוח התור (`/queue`) — המסך המרכזי; (2) שורת-מטופל מורחבת; (3) פאנל/מודאל ניהול פעילויות+אצל-מי (נפתח מהשורה).

### מסך: לוח התור (הרחבת `QueuePage`)
- **מטרה:** מצב המחלקה במבט אחד.
- **נראה לתפקידים:** כל הצוות (קליני רואה פעולות; קבלה רואה לקריאה + שחרור).
- **שורת מטופל — תצוגה:** `מס׳ תור · שם · גיל · [באדג׳ שלב] · אצל: [תחנה] · [צ'יפ פעילות⏳ …] · [דגלים] · המתנה · מחלקה · פעולות`.
  - **באדג׳ שלב** — הסטטוס הראשי (ציר 1), צבע מהרמפה המעומעמת.
  - **"אצל: …"** — ציר 3, בולט (זה מה שזז הכי הרבה).
  - **צ'יפים** — פעילות פתוחה + אייקון מצב (הוזמן/בביצוע/תוצאה). פעילות שהושלמה נעלמת מהשורה (או מסומנת ✓).
- **פעולות (תלויות שלב+תפקיד):** קדם שלב · החלף "אצל מי" · הוסף/עדכן פעילות · חתום (רופא) · שחרר (קבלה).
- **מצבים:** ריק ("אין מטופלים בתור"), טעינה, אין-הרשאה (קבלה לא רואה פעולות קליניות).

> **שתי האפשרויות להצגה למנהל (לאישור §10):** נבנה **שתי וריאציות של ייצוג השורה** — נגדיר אותן יחד — כדי שהמנהל יבחר.

## 7. לוגיקה מיוחדת ומנגנונים

### מנגנון: קידום שלב מול ניהול פעילויות/אצל-מי
- **כלל:** רוב הפעולות הקליניות **אינן** משנות שלב. שינוי שלב הוא אירוע נדיר ומכוון (טריאז׳ הסתיים → "ממתין לרופא"; רופא חתם → "סיים טיפול"). תנועה רופא↔אחות = שינוי `CurrentStation` בלבד. "ממתין לתוצאה" = מצב פעילות, לא שלב.
- **דוגמה עובדת:** ראה §3.
- **כשל/שימוש לרעה:** חזרה לשלב אחורה — חסומה לצוות רגיל, מותרת למנהל משמרת/אדמין בלבד, מתועדת ב-audit.

### מנגנון: ריבוי-עורכים בזמן אמת
עדכוני שלב/פעילות/אצל-מי משודרים ב-SignalR (`QueueUpdate` קיים) לכל הצופים בלוח.

## 8. דרישות לא-פונקציונליות

### אבטחה והרשאות (מטריצה — טיוטה)
| פעולה | קבלה | אחות | רופא | מנהל משמרת | אדמין |
|-------|------|------|------|-----------|-------|
| קידום שלב טריאז׳ | ✗ | ✓ | ✓ | ✓ | ✓ |
| התחל טיפול / הזמן פעילות | ✗ | חלקי* | ✓ | ✓ | ✓ |
| עדכן מצב פעילות (בוצע/תוצאה) | ✗ | ✓ | ✓ | ✓ | ✓ |
| החלף "אצל מי" | ✗ | ✓ | ✓ | ✓ | ✓ |
| חתימה → סיים טיפול | ✗ | ✗ | ✓ | ✗ | ✗ |
| שחרור | ✓ | ✗ | ✗ | ✓ | ✓ |
| חזרה לשלב אחורה | ✗ | ✗ | ✗ | ✓ | ✓ |

*חלקי = אחות יכולה לפתוח פעילויות סיעודיות (מתן/השגחה), לא הזמנות רפואיות.

### פרטיות ורגולציה
PHI קליני — כל פעולה נרשמת ל-`AuditService` (מי/מה/מתי/IP). תואם לתקן הקיים (docs/security). פעילויות/שלב = מטא-דאטה קלינית; חשיפה לפי "צורך לדעת" כמו היום.

### i18n / RTL
עברית RTL מלא; באדג׳ים/צ'יפים מיושרים-לימין; טבלאות ממורכזות (לפי העיצוב הקיים).

### תצפיתיות
זמן-במחלקה לכל מטופל; דגל "חריגת המתנה" (קיים בלוח). אפשר בעתיד: מדדי תפוקה (LWBS%, זמן-לרופא).

## 9. אילוצים, הנחות ותלויות
- **אילוץ:** On-Prem, ללא אינטגרציות חיצוניות — תוצאות מוזנות ידנית.
- **הנחה:** קטלוג התחנות הקיים (`STATION_OPTIONS`) הוא הבסיס לסוגי הפעילויות/התחנות.
- **תלות:** מיגרציית EF להוספת `VisitPhase`/`VisitActivity` + מיגרציית נתונים מ-`VisitStatus`.

## 10. שאלות פתוחות וסיכונים

**שאלות פתוחות:**
- [ ] גרנולריות פעילות — **3 מצבים** (מוצע) מול 2. בעלים: המשתמש.
- [ ] סיומים — `שוחרר` + `הופנה/אושפז` (מוצע) + `LWBS`? בעלים: המשתמש.
- [ ] "דיספוזיציה" — מקופלת ל"בטיפול" (מוצע) מול שלב נפרד. בעלים: המשתמש.
- [ ] **שתי האפשרויות להצגה למנהל** — מה בדיוק הן (שתי וריאציות שורה? מודל עשיר מול פשוט?). בעלים: המשתמש.
- [ ] `VisitPhase` מחליף את `VisitStatus`, או חי לצידו ונגזר ממנו?

**סיכונים:**
- מורכבות עודפת לצוות → מיטיגציה: השלב נשאר פשוט; הפירוט אופציונלי בצ'יפים.
- מיגרציית סטטוסים קיימים → מיטיגציה: מיפוי 1:1 מוגדר מראש (§3), בדיקה על נתוני הדמו.
