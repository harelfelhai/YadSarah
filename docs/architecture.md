# תכנון מערכת ניהול טיפול — יד שרה, רפואה דחופה

## רקע (Context)

יד שרה — מחלקת רפואה דחופה — זקוקה למערכת חדשה שתחליף את הקיימת, ותנהל את כל מסע
המטופל: מרגע ההגעה לקבלה, דרך התור והטיפול, ועד לשחרור.

**שתי דרישות-על שמכתיבות את כל הארכיטקטורה:**
1. **אבטחה** בתקני ארגון בריאות מחמירים (פירוט מדויק יגיע מהמשתמש בהמשך).
2. **אי-תלות באינטרנט** — המערכת חיה כולה ברשת פנימית (On-Prem LAN); האינטרנט אינו
   בנתיב הקריטי של אף פעולה.

**דרישות מהותיות נוספות שעלו:**
- עריכה מרובת-משתמשים של אותו טופס (רופא + אחות) עם מנגנון שמונע דריסה של אותו שדה
  בו-זמנית.
- ייצוא נתוני הטיפול (טופס הרופא) בסוף הטיפול.
- ישויות הליבה: **משתמשים** (תפקידים → הרשאות), **לקוחות/מטופלים**, **ביקורים**,
  **טפסים רפואיים** (ביקור אחד → מספר טפסים לפי מחלקות).

**החלטות שהתקבלו עם המשתמש:**
- Stack: סטנדרטים מודרניים/מקצועיים, גמיש, בעל יכולת התממשקות (interoperability),
  ועמיד ל-scale גדול.
- אימות: חשבונות מקומיים במערכת (ללא תלות ב-AD ארגוני).
- פריסה: אתר יחיד, שרת מקומי אחד.
- עמידות: שרת מרכזי מספיק כרגע; לתכנן כך שהוספת גיבוי/HA בעתיד לא תדרוש שינוי מורכב.

**עיצוב:** נקי, מודרני, מתאים לעסק רפואי. עברית/RTL כשפת ברירת מחדל.

> הערה: זהו מסמך **תכנון ארכיטקטוני בלבד**. השדות המדויקים בכל מסך (קבלה, טפסים לפי מחלקה),
> מסמך תקני האבטחה, ופורמט הייצוא — יסופקו בהמשך ע"י המשתמש (ראו "פריטים פתוחים").
> **אין לממש דבר** (כולל זרימות, אבטחה, מבנה נתונים) עד שהמשתמש יגדיר אותו במפורש —
> ללא הנחות/מימושים "רנדומליים".

## ארכיטקטורה — מבט-על

ארכיטקטורת 3 שכבות, **API-first**, רצה במלואה ברשת הפנימית:

```
┌─────────────────────────────────────────────────────────────┐
│  תחנות עבודה ברשת הפנימית (קבלה / אחות / רופא / לוח תצוגה)     │
│  דפדפן → React + TypeScript SPA  (RTL, נקי ומודרני)          │
└───────────────┬───────────────────────────┬─────────────────┘
                │ HTTPS (TLS פנימי)          │ WebSocket (SignalR)
┌───────────────▼───────────────────────────▼─────────────────┐
│  שרת אפליקציה On-Prem  —  ASP.NET Core                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────────────┐  │
│  │ Auth/RBAC│ │ Patients │ │  Visits/ │ │ Medical Forms  │  │
│  │ + Audit  │ │          │ │  Queue   │ │ + Concurrency  │  │
│  └──────────┘ └──────────┘ └──────────┘ └────────────────┘  │
│  ┌──────────────────────┐ ┌──────────────────────────────┐  │
│  │ Export (PDF + FHIR)  │ │ Integration / API (OpenAPI)  │  │
│  └──────────────────────┘ └──────────────────────────────┘  │
└───────────────┬─────────────────────────────────────────────┘
                │ TLS
┌───────────────▼─────────────────────────────────────────────┐
│  PostgreSQL (On-Prem)  —  הצפנה בנייחה, גיבויים מוצפנים       │
│  מתוכנן לתוספת replica/standby בעתיד ללא שינוי אפליקטיבי      │
└─────────────────────────────────────────────────────────────┘
```

**בחירת טכנולוגיות ונימוק:**
- **Frontend: React + TypeScript** — אקוסיסטם חזק ל-RTL/עברית, רכיבי UI נקיים ומודרניים,
  נגישות. ספריית UI עקבית (למשל Mantine/MUI עם ערכת נושא רפואית מותאמת).
- **Backend: ASP.NET Core** — enterprise-grade, מצוין ל-On-Prem על Windows, חינמי/קוד-פתוח,
  ביצועים ו-scale גבוהים, **SignalR** מובנה לעריכה בזמן-אמת.
- **DB: PostgreSQL** — חינמי, חזק, תמיכה ב-row-versioning (xmin) לבקרת מקביליות, הצפנה,
  ו-replication להרחבה עתידית.
- **Real-time: SignalR (WebSocket)** — נוכחות (presence) ונעילות שדה/מקטע בזמן-אמת.
- **API contract: OpenAPI/Swagger** — לתמיכה ב-interoperability עם מערכות אחרות.

מבנה הקוד יהיה **modular monolith** (מודולים מופרדים בתוך שירות אחד) — פשוט לפריסה ולתפעול
On-Prem באתר יחיד, אך עם גבולות מודולריים נקיים שמאפשרים פיצול לשירותים אם ה-scale ידרוש.

## מודל הנתונים (Domain Model)

ישויות עיקריות והקשרים ביניהן:

- **User** — משתמש מערכת (חשבון מקומי). שדות: שם, מזהה, hash סיסמה, סטטוס, role(s).
- **Role / Permission** — RBAC. תפקידים: קבלה, אחות, רופא, מנהל מערכת (ועוד לפי הצורך).
  הרשאות נגזרות מתפקיד; כולל **הרשאה ברמת שדה/מקטע** (איזה תפקיד רואה/עורך מה).
- **Patient (לקוח)** — מטופל. **המפתח הוא מזהה פנימי (internal ID)** שהמערכת מנפיקה — לא ת"ז.
  ת"ז הוא **שדה רגיל ואופציונלי**, שכן לעיתים אין ת"ז אלא זיהוי אחר (דרכון/זיהוי חלופי);
  ייתכן תמיכה במספר סוגי זיהוי. כולל פרטים אישיים, קופ"ח, טלפון וכו'. מטופל אחד →
  מספר ביקורים לאורך זמן (היסטוריה).
- **Visit (ביקור)** — מופע טיפול בודד. מחזיק את **מכונת המצבים** של התור/הטיפול. ביקור אחד →
  מספר טפסים.
- **MedicalForm (טופס רפואי)** — טופס לפי מחלקה/דיסציפלינה, משויך לביקור. מחולק ל**מקטעים**
  (sections) ו**שדות** — כולל תרופות ואבחנות, שהם **שדות בתוך הטופס** (לא ישויות/טבלאות
  נפרדות). כולל גרסה (row version) לבקרת מקביליות.
- **AuditLog** — יומן בלתי-ניתן-לשינוי של כל גישה/שינוי ל-PHI (מי, מה, מתי, ערך-קודם/חדש).
- **FormLock** — רשומת נעילה זמנית של מקטע/שדה (משתמש, זמן, תוקף) לניהול המקביליות.

**מכונת המצבים של הביקור (Queue/Visit status):**
```
ממתין (Waiting) → נקרא (Called) → בטיפול (InTreatment) → הושלם/שוחרר (Discharged)
```
מעברים מתועדים ב-Audit; "נקרא" מופעל בכפתור הקריאה; "בטיפול" — לפי אפיון: לחיצה ידנית או
אוטומטי כשרופא/אחות פותחים את הטופס (יוכרע עם השדות המדויקים).

## מודולים / מסכים

1. **קבלה (Reception)** — קליטת מטופל (חיפוש לפי ת"ז / יצירת חדש), פתיחת ביקור, הכנסה לתור.
2. **תור (Queue)** — רשימת ממתינים לפי מספרים, כפתור "קרא למטופל" (→ סטטוס "נקרא"),
   מעבר ל"בטיפול", ולוח תצוגה (display board) להקרנה.
3. **טופס טיפול (Treatment Form)** — טופס רב-מקטעי, **עריכה מרובת-משתמשים עם נעילה**,
   הזנת אבחנות ותרופות, צירוף טפסים נוספים לפי מחלקה.
4. **שחרור וייצוא (Discharge & Export)** — סיום טיפול, ייצוא טופס הרופא (PDF + מובנה).
5. **ניהול (Admin)** — משתמשים/תפקידים/הרשאות, הגדרות מערכת, צפייה ביומני ביקורת.

## עיצוב המקביליות (דרישת-מפתח: מניעת דריסה בו-זמנית)

שילוב של **נעילה רכה בזמן-אמת** + **בקרת מקביליות אופטימית** כגיבוי:

1. **Presence** — בפתיחת טופס, הקליינט מצטרף לקבוצת SignalR של הביקור/טופס; כולם רואים מי
   עוד נמצא בטופס.
2. **נעילת מקטע/שדה** — כשמשתמש מתחיל לערוך מקטע (ברירת מחדל; אופציה לרמת שדה), הקליינט
   מבקש נעילה מהשרת. אם פנוי — השרת מעניק ומשדר "מקטע X ננעל ע"י משתמש Y"; אצל האחרים
   המקטע מוצג לקריאה-בלבד עם חיווי ברור.
3. **שחרור ושידור** — בעת blur/שמירה/timeout הנעילה משתחררת, והערך המעודכן משודר לכולם
   בזמן-אמת (live).
4. **גיבוי מקביליות (backstop)** — לכל רשומה יש version/row-version (xmin ב-Postgres או עמודת
   version); בשמירה נבדקת הגרסה כדי למנוע lost-update גם אם נעקפה הנעילה.
5. **עמידות לקריסות** — heartbeat לנעילות; נעילה של קליינט שקרס פגה אוטומטית בתום timeout.

החלטה לפינון בהמשך: גרנולריות (מקטע מול שדה), שמירה אוטומטית מול ידנית, ו-UX לקונפליקט.

## עיצוב האבטחה (דרישת-מפתח)

> התקנים המדויקים יסופקו בהמשך; התכנון בנוי **גמיש ומתאים-לקונפיגורציה** כדי להיישר לדרישה
> הסופית. מסגרות יעד: תקנות הגנת הפרטיות (אבטחת מידע) התשע"ז-2017 — מאגר **ברמת אבטחה
> גבוהה**, חוזרי אבטחת מידע/סייבר של משרד הבריאות, ויישור ל-ISO 27799 / 27001.

- **אימות (AuthN):** חשבונות מקומיים, hashing חזק (Argon2id/bcrypt), מדיניות סיסמאות,
  **MFA אופציונלי**, פקיעת סשן, ונעילת מסך אוטומטית (auto-lock) לתחנות.
- **הרשאות (AuthZ):** RBAC עם least-privilege; **נראות/עריכה ברמת שדה ומקטע** לפי תפקיד.
- **הצפנה בתעבורה:** TLS גם ברשת הפנימית (CA פנימי/ארגוני).
- **הצפנה בנייחה:** הצפנת דיסק (BitLocker) + הצפנה ברמת ה-DB; ניהול מפתחות מוגדר.
- **יומן ביקורת בלתי-ניתן-לשינוי:** כל גישה/שינוי ל-PHI מתועד (מי/מתי/מה/לפני-אחרי).
- **גיבויים מוצפנים** ותהליך שחזור (DR) — ברשת הפנימית.
- **מינימיזציית נתונים** והקשחת תחנות; כל ה-secrets מנוהלים מחוץ לקוד.

## אי-תלות באינטרנט / On-Prem

- אין שום קריאת ענן בנתיב הקריטי. הכול חי ברשת הפנימית.
- שירותי תשתית מקומיים (NTP/DNS פנימיים לפי הצורך).
- עדכוני גרסה נמסרים כ**חבילת offline** פנימית — לא משיכה אוטומטית מהאינטרנט.
- ידידותי ל-air-gap: ניתן להריץ ללא חיבור חיצוני כלל.

## Interoperability & Scale (לפי דרישת המשתמש)

- **API-first** עם OpenAPI — בסיס נקי להתממשקות עם מערכות אחרות.
- **ייצוא/התממשקות בתקן בריאות:** שכבת ייצוא ל-**FHIR/HL7** (לצד PDF) — תקן ה-interoperability
  המקובל בבריאות; מאפשר חיבור עתידי למערכות חיצוניות.
- **מוכנות ל-scale ול-HA:** שכבות מופרדות, שרת אפליקציה נטול-מצב (stateless) ככל הניתן,
  ו-DB מתוכנן כך שתוספת replica/standby + load-balancing + גיבוי תיעשה ללא ארכיטקטורה-מחדש.

## תוכנית מסירה מדורגת (Phased delivery)

- **Phase 0 — תשתית:** scaffolding (Server+Client), CI מקומי, Auth + RBAC + AuditLog בסיסי,
  TLS פנימי. *(לא תלוי בשדות המדויקים — אפשר להתחיל מיד אחרי אישור.)*
- **Phase 1 — מטופל + ביקור + תור:** רישום קבלה, מכונת מצבים, כפתור קריאה, לוח תצוגה.
- **Phase 2 — טפסים רפואיים + מקביליות:** טופס רב-מקטעי, SignalR presence + נעילות, תרופות/אבחנות.
- **Phase 3 — שחרור + ייצוא:** סיום טיפול, ייצוא PDF + FHIR/HL7.
- **Phase 4 — הקשחה ועמידות:** גיבויים מוצפנים, מוכנות HA/replica, ביקורת אבטחה, בדיקות עומס.

## אימות ובדיקה (Verification)

- **End-to-end ידני:** הרצת השרת והקליינט ברשת מקומית; מעבר מלא מטופל קבלה → תור → קריאה →
  טיפול → שחרור → ייצוא.
- **תרחיש מקביליות:** שני דפדפנים (רופא+אחות) על אותו טופס — אימות שנעילת מקטע מונעת דריסה,
  והעדכון משודר בזמן-אמת.
- **בדיקות אוטומטיות:** יחידה ל-state machine/RBAC/נעילות; אינטגרציה ל-API; e2e לזרימות
  המסך המרכזיות.
- **אבטחה:** אימות יומן ביקורת לכל שינוי PHI, בדיקת TLS, ובדיקות הרשאה לפי תפקיד.
- **Offline:** ניתוק האינטרנט (לא הרשת הפנימית) ואימות שכל הזרימות עובדות.

## פריטים פתוחים — ממתינים לקלט מהמשתמש

1. **שדות מדויקים** במסך הקבלה ובכל טופס רפואי (לפי מחלקה) + תמונות מהמערכת הקיימת.
2. **מסמך תקני האבטחה** המדויקים שאליהם צריך להיישר.
3. **פורמט הייצוא** הנדרש בסוף הטיפול (PDF? FHIR? XML? יעד התממשקות?).
4. **רשימת התפקידים** המלאה ומטריצת ההרשאות (מי רואה/עורך מה).
5. גרנולריות הנעילה (מקטע/שדה) והתנהגות מעבר ל"בטיפול" (ידני/אוטומטי).
